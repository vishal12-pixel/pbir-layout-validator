# Contract: `conf.md` Rules File Format

This is the on-disk format produced by `learn` and consumed by `validate` and `fix`.
The format is human-readable Markdown so a developer can sanity-check or hand-edit it
without running the tool.

## Grammar

```text
file        := (line NEWLINE)*
line        := blank | comment | header | rule
blank       := ws*
comment     := ws* "#" any*
header      := ws* "#"+ ws+ any+        # treated identically to a comment by the parser
rule        := ws* from_type ws* "->" ws* to_type ws* ":" ws* int "px" ws*
from_type   := identifier
to_type     := identifier
identifier  := [A-Za-z_][A-Za-z0-9_-]*  # PBIR visual.visualType values are this shape
int         := -?[0-9]+
ws          := " " | "\t"
```

## Semantics

- A non-blank, non-comment line is a **rule**.
- The parser is **case-sensitive** on `from_type` and `to_type` (per spec Assumptions:
  no case-folding or synonym mapping).
- `gap_px` is parsed as a Python `int`. Negative values are accepted by the parser but
  flagged as a warning: gaps are conventionally non-negative; a negative gap means the
  rows overlap, which is unusual but legal.
- Duplicate rules for the same `(from_type, to_type)` pair: parser warns and keeps the
  **first** occurrence (so a developer can hand-override by adding a line at the top).
- Unknown trailing tokens on a rule line → warn and skip the line; do not abort the run.

## Canonical writer output

`learner.write_conf(path, rules, reference_page_name)` produces this exact shape:

```markdown
# PBIR Layout spacing rules
# Generated by pbir_validator from page: <reference_page_name>
# Format: <from_type> -> <to_type>: <gap>px
# Edit by hand if needed; comments start with #.

card -> actionButton: 9px
actionButton -> tableEx: 16px
tableEx -> shape: 12px
```

Rules are emitted sorted by `(from_type, to_type)` ascending so that re-running `learn`
on an unchanged page produces a byte-identical file.

## Invariants enforced by the parser

- File missing → exit code `5` with message `conf.md not found at <path>`.
- File present but contains zero rules → exit code `5` with message
  `conf.md at <path> contains no rules`.
- A rule that fails the grammar above → warn (with line number) and skip; if zero rules
  remain after skipping, treat as the previous case.

## Reference: example with conflicting source page

When the reference page contained conflicting gaps for the same pair, the writer still
emits one canonical rule per pair (most-frequent, ties → smallest) and prepends a
warning comment:

```markdown
# PBIR Layout spacing rules
# Generated by pbir_validator from page: Sales Overview
# WARNING: page contained conflicting gaps for: card -> actionButton (saw 9, 9, 11)
#          chose 9 (most frequent).

card -> actionButton: 9px
...
```

The `WARNING:` comments are informational only; the parser ignores them on read.
